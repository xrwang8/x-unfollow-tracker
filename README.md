# X 取关追踪器

追踪 X（Twitter）关注者变化，找出取关你的用户。

## 为什么选择这个扩展？

| 功能 | 其他扩展 | 本扩展 |
|------|---------|--------|
| **检测速度** | 8-10 分钟 | **3-5 秒** ⚡ |
| **数据来源** | 页面滚动抓取 | **X 官方 API** |
| **历史追踪** | ❌ | ✅ 30 天快照对比 |
| **后台运行** | ❌ 必须盯着页面 | ✅ |
| **数据导出** | ❌ | ✅ CSV 格式 |
| **Manifest** | V2（淘汰中） | **V3** |
| **抗失效** | ❌ HTML 改就挂 | ✅ API 降级方案 |

**速度对比（500 关注者）**：
- 其他扩展：需滚动 500 次，每次等 1 秒 = **8-10 分钟** 😱
- 本扩展：3 次 API 请求 = **3-5 秒** ⚡（**快 150 倍**）

## 功能

- ⚡ **极速检查** — 500 人仅需 3-5 秒（API 方式，比竞品快 150 倍）
- 📊 **双模式分析**
  - 谁取关了你（对比历史快照）
  - 未回关列表（你关注但对方没关注你）
- 📸 **快照管理** — 每次检查保存快照，自动保留 30 天
- 🔍 **历史对比** — 查看过去 7 天 / 30 天的变化
- 💾 **CSV 导出** — 一键导出分析结果到 Excel
- 🛡️ **API 降级** — 接口失效时自动切换到备用方案
- 🔒 **隐私安全** — 所有数据存本地，不经过任何第三方服务器

## 工作原理

用你的 X 登录态（页面 `ct0` cookie + 站点公开 bearer）调用 X 的 `followers/list.json` 接口，分页拉取关注者列表。限速 1s/页 + 抖动 + 退避，避免触发 X 反滥用。

**API 降级方案**：如果 X 改了接口导致 API 失效（401/403/404），扩展会自动切换到备用方案（DOM 页面滚动抓取），虽然慢但仍能工作。你会在进度提示里看到 `[页面滚动]` 标记。

## 使用

### 快速开始

1. **确保已登录 x.com**
2. **点击扩展图标**打开 popup
3. **首次检查**：
   - 点"检查关注者"→ 拉取并保存关注者快照
   - 点"检查你的关注"→ 拉取并保存关注快照
4. **等待几天/几周后再次检查**，保存新快照
5. **分析变化**：
   - 选择分析模式（谁取关了你 / 未回关列表）
   - 选择对比范围（过去 7 天 / 30 天）
   - 点"查看结果"
6. **导出数据**：点结果列表右上角"导出 CSV"按钮

### 分析模式说明

**谁取关了你**
- 需要：至少 2 个"关注者"快照
- 对比：旧快照 vs 新快照
- 结果：在旧快照里有、在新快照里没有的用户

**未回关列表**
- 需要：至少 1 个"关注者"快照 + 1 个"你的关注"快照
- 对比：你的关注 vs 关注者
- 结果：你关注了但对方没关注你的用户

## 已知限制

- **API 依赖**：依赖 X 内部 API（可能失效，但会自动降级到备用方案）
- **首次检查耗时**：500 人约 3-5 秒（API）或 12-15 分钟（备用方案）
- **需要登录**：必须在 x.com 登录后才能使用
- **快照对比**：至少需要 2 个快照才能分析变化
- **数据本地化**：卸载扩展会丢失所有快照

## 常见问题

### Q: 为什么进度显示 `[页面滚动]`？
A: X 的 API 失效了，扩展自动切换到备用方案（DOM 抓取）。虽然慢但仍能工作。

### Q: 显示"Rate limit (429)"怎么办？
A: X 临时限速，等待 1-2 分钟后重试。已拉取的部分数据会保存。

### Q: 未回关列表为什么需要两次检查？
A: 需要分别获取"关注者"和"你的关注"两份数据才能对比。

### Q: 可以追踪更久的历史吗？
A: 快照自动保留 30 天，超期会自动清理。建议每周检查一次。
